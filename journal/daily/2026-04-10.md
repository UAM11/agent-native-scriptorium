## Frictions

- “数据飞轮”“eval 飞轮”“RFT”“Agentic RL”在网上经常被混用，很容易把成熟度不同的事情说成同一件事
- 很多真实案例会公开结果和原则，但不会公开完整内部 schema，所以整理时需要明确区分“官方事实”和“工程抽象”
- 如果想把飞轮讲得既准确又可落地，必须同时覆盖产品、评估、训练和治理，而不能只谈模型

## Insights

- 对大多数垂直 agent 来说，最先跑起来的应该是 `trace -> review -> eval -> workflow update`，而不是直接 RL
- 客服、法务、coding 三类场景的共同点都是“需要可靠反馈”，但它们最自然的反馈形式并不一样
- coding agent 之所以特别适合 Agentic RL，不是因为“更酷”，而是因为环境反馈天然更清晰
- 真实案例最值得学的不是某个单点指标，而是它们如何把生产数据变成持续迭代机制
