## 背景

随着资料开始同时包含可公开分享的内容与不适合公开的内容，单仓库模式已经不够稳妥。双仓结构的目标，是让知识既能被分享，又不会因为过早公开而泄露隐私、暴露未成熟设计，或把强个人上下文直接暴露出去。

## 角色分工

- `agent-native-scriptorium`：公开展架。只收录可公开、可复用、已脱敏的资料。
- `agent-native-vault`：私有密库。用于存放未公开、敏感、半成品、带有强个人上下文的资料。

## 路由规则

- 可公开分享、已完成脱敏、适合被未来他人或 agent 直接复用的内容，进入 `scriptorium`。
- 仍在探索中、带有隐私、包含内部设计细节、尚未准备公开的内容，先进入 `vault`。
- 判断不准时，默认先放 `vault`。
- 任何密钥、令牌、密码、cookie、身份证件、银行卡信息等高风险秘密，不进入任一仓库。

## 私仓晋升公仓的检查清单

- 这份内容是否已经能独立成立，而不是原始聊天碎片？
- 是否移除了私人身份信息、敏感项目细节、未公开设计或不适合外泄的上下文？
- 是否已经提炼出结论、步骤、验证方式和适用边界？
- 是否可以被公开读者或未来 agent 直接理解和复用？
- 是否需要在公开版本中补充“这是脱敏后的整理稿”之类的说明？

## 操作步骤

1. 新内容产生后，先判断它是适合公开，还是应该先进入私仓。
2. 如果内容涉及隐私、不确定、半成品或内部设计，先写入 `agent-native-vault`。
3. 如果内容已经适合公开，直接整理后放入 `agent-native-scriptorium`。
4. 当私仓内容成熟后，重新写成面向复用的公开版本，而不是机械复制原稿。
5. 公共版本落库后，根据价值同步更新 `README.md` 与 `CHANGELOG.md`。
6. 如有必要，在公开版本中保留与私密原稿的主题承接，但不要泄露私密上下文。

## 验证

- 于 2026-03-23 建立并验证了 `agent-native-scriptorium` 与 `agent-native-vault` 的双仓目录。
- 私仓已包含一条仅存在于 `vault` 的测试 idea，用于验证“公共内容共享、私密内容隔离”的基本形态。

## 注意事项

- 私仓不是“随便堆聊天记录”的地方，仍然要尽量写成可检索、可演化的条目。
- 公仓不是“私仓的镜像”，而是筛选、提炼、脱敏后的公共知识展架。
- 如果同一主题既有私密草稿又有公开版本，优先让两者在结论层承接，而不是全文同步。
